🔷 P1: Wireless vs Mobile Computing

Point
Wireless Computing dan Mobile Computing berbeda secara fundamental: wireless membahas media komunikasi tanpa kabel, sedangkan mobile membahas kemampuan entitas (user/device/service) untuk tetap beroperasi saat berpindah lokasi.

Reason
Wireless hanya menyelesaikan layer fisik/link (bagaimana data dikirim tanpa kabel), sementara mobile menuntut kontinuitas layanan di tengah perubahan topologi jaringan (handover, perubahan IP, fluktuasi sinyal). Artinya, mobile melibatkan tantangan lintas layer: jaringan, transport, hingga aplikasi.

Example

Wireless saja: Smart TV terhubung WiFi (tidak berpindah).
Mobile saja: Laptop dipindahkan tanpa koneksi jaringan.
Keduanya: Smartphone streaming saat berpindah dari WiFi ke 4G (terjadi handover dan adaptasi bitrate).

Point (penegasan)
Mobile computing meng-extend wireless dengan aspek mobilitas dan kontinuitas layanan; karena itu problemnya lebih kompleks (handover, latency, session persistence).

🔷 P2: Device/User/Service Mobility + Handover

Point
Handover adalah mekanisme mempertahankan sesi komunikasi saat perangkat berpindah antar sel/AP, dengan dua pendekatan utama: Hard Handoff dan Soft Handoff.

Reason
Mobilitas menyebabkan perubahan kualitas sinyal dan titik akses. Tanpa handover yang tepat, sesi akan putus.

Hard Handoff (Break-Before-Make): koneksi lama diputus dulu, baru sambung ke yang baru → ada gap.
Soft Handoff (Make-Before-Break): koneksi baru dibangun sebelum yang lama dilepas → overlap, lebih seamless.

Example
Pada voice call, jeda beberapa ratus milidetik saja terasa sebagai “putus-putus”.

Hard handoff → drop audio saat pindah BTS.
Soft handoff → audio tetap kontinu karena dua koneksi overlap sementara.

Point (penegasan)
Untuk voice/real-time, Soft Handoff lebih unggul pada dimensi latency & continuity, karena menghindari interupsi sesi.

🔷 P3: WPAN/WLAN/WMAN/WWAN + IEEE & Mode

Point
Klasifikasi jaringan nirkabel dibedakan oleh cakupan dan standar: WPAN (802.15), WLAN (802.11), WMAN (802.16), WWAN (cellular), serta mode operasi Ad-Hoc vs Infrastructure.

Reason
Setiap kategori mengoptimalkan trade-off berbeda (range, bandwidth, power). Mode operasi menentukan bagaimana routing dan manajemen koneksi dilakukan.

Ad-Hoc: peer-to-peer, tanpa koordinasi pusat → fleksibel tapi kurang stabil.
Infrastructure: melalui AP/BTS → manajemen terpusat, QoS lebih baik.

Example

Bluetooth (802.15) untuk perangkat dekat dan hemat energi.
WiFi (802.11) untuk LAN berkecepatan tinggi.
4G/5G (WWAN) untuk mobilitas luas dengan dukungan handover kompleks.

Point (penegasan)
Perbedaan standar dan mode ini menentukan latency, bandwidth, dan reliability, yang langsung mempengaruhi desain OS dan aplikasi di atasnya.

🔷 P4: Arsitektur 4 Lapisan OS Mobile + Sandbox

Point
OS Mobile (contoh: Android) terdiri dari 4 lapisan: Kernel → Libraries/Runtime → Application Framework → Applications, dengan sandboxing ketat sebagai mekanisme keamanan utama.

Reason
Perangkat mobile memiliki resource terbatas dan data sensitif (kamera, lokasi).

Kernel: manajemen proses, memori, driver.
Runtime (ART): eksekusi aplikasi.
Framework: API tinggi (Activity Manager, dll).
Sandbox: tiap app berjalan dengan UID terpisah → isolasi.

Example
Aplikasi A tidak bisa langsung akses data Aplikasi B tanpa permission & IPC. Bahkan akses kamera/GPS harus melalui API terkontrol.

Point (penegasan)
Sandboxing pada mobile lebih ketat dari desktop karena kombinasi faktor: keamanan data pribadi + keterbatasan resource + model distribusi aplikasi yang terbuka.

🔷 P5: LMK + IPC + Intent (Hierarki Proses Android)

Point
Android mengelola resource dengan hierarki prioritas proses dan mekanisme Low Memory Killer (LMK), serta komunikasi antar komponen via IPC & Intent.

Reason
Karena RAM terbatas dan tidak mengandalkan swap berat, sistem harus agresif membebaskan memori.
Hierarki proses:

Foreground
Visible
Service
Background
Empty

LMK akan membunuh dari prioritas terendah.

Example
Saat user membuka aplikasi berat:

App background bisa dibunuh.
Jika app tidak simpan state → data hilang.

Intent:

Explicit → target jelas
Implicit → sistem pilih handler

Point (penegasan)
Desain aplikasi harus lifecycle-aware karena OS bisa menghentikan proses kapan saja; ini bukan bug, tapi desain sistem.

🔷 P6: Native vs Hybrid vs PWA (Dimensi A,P,B,D,P)
📊 Tabel Komparasi
| Dimensi                | Native                     | Hybrid                 | PWA                |
| ---------------------- | -------------------------- | ---------------------- | ------------------ |
| **Akses Hardware (A)** | Full (NFC, Camera, ARCore) | Plugin (Capacitor, RN) | Web API terbatas   |
| **Performa (P)**       | Tinggi (compiled)          | Menengah               | Rendah–menengah    |
| **Biaya Tim (B)**      | Tinggi (multi-OS)          | Menengah               | Rendah             |
| **Distribusi (D)**     | App Store                  | App Store              | Browser (URL)      |
| **Pilihan Tepat (P)**  | AR/VR, game                | POS, enterprise app    | Dashboard, katalog |

Point
Tidak ada pendekatan yang “paling baik”; pemilihan harus berbasis dimensi kebutuhan (A,P,B,D,P).

Reason
Setiap pendekatan mengoptimalkan trade-off berbeda:

Native → kontrol penuh & performa
Hybrid → efisiensi dev + cukup akses hardware
PWA → distribusi cepat & biaya rendah

Example

AR/VR → butuh low-level hardware → Native
POS → butuh stabil + hardware sedang → Hybrid
Sistem inventaris berbasis web → PWA cukup

Point (penegasan)
Keputusan arsitektur harus menyebut dimensi spesifik, bukan klaim umum seperti “lebih baik”.
